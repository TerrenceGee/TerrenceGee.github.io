数据持久化层场景实战​

1.冷热分离​

将常用的“热数据”和不常使用的“冷”数据分开存储

锁的机制、批量处理、失败重试的数据一致性问题

Sass 平台，用户无需购买和维护硬件或软件，只需支付使用服务的费用。常见的 Saas 应用有，Google Workspace、Microsoft 365等等

1.1 工单邮件案例​

背景：

itemdesc
工单表30_000_000
工单处理记录表150_000_000
工单表增长速度100_000/day

已经通过 优化表结构、业务代码、索引、SQL 语句等办法，将数据支撑做到了 30_000_000的数据量。

1.2 数据库分区​

数据库分区：将表的数据均衡分配到不同的硬盘、系统或不同的服务器存储介质中，实际上还是一张表。

1.3 冷热分离​

冷热分离就是在处理数据时将数据库分为冷库和热库，冷库存放那些走到终态、不常使用的数据，热库存放还需要修改，经常使用的数据。
使用冷热分离的情形：

  1. 数据走到终态后只有读没有写的需求，如订单完结状态。
  2. 用户能接受数据分开查询，比如有些电商默认只允许查询3个月内的订单数据。如果要查询3个月前的订单，还需要访问其他的页面。

1.4 冷热分离实现思路：冷热分离都使用 mysql​

遇到的问题：

  1. 如何判断一个数据是冷数据还是热数据？
  2. 如何触发冷热数据分离？
  3. 如何实现冷热数据分离？
  4. 如何使用冷热数据？
  5. 历史数据如何迁移？

判断一个数据冷热数据：可以使用表中某个字段，可以是时间、状态，也可以结合时间和状态
如何触发冷热数据分离：\

  • 每次涉及到订单修改时，判断是否进行冷热分离数据处理；根据状态区分冷热数据时可以使用。
  • 可以通过监听数据库变更日志 binlog + 另起服务监控 + 消息队列；业务代码复杂不适合修改，并且不根据时间区分冷热数据
  • 定时扫描数据库；按照时间区分冷热数据时使用。

如何分离冷热数据？

其实质是删除热数据，并插入对应的冷数据。这是一个事务特性的操作，但是因为冷热库分离，所以无法依赖数据库的事务特性。解决办法是最终一致性的：

  1. 将热库待删除的数据添加一个标识WaitForMove，然后将其删除。
  2. 找出所有待迁移的数据（通过 WaitForMove）。这一步保证之前的数据迁移失败，可以进行补偿操作
  3. 在冷库中插入一份数据，这个数据要有幂等性判断。这一步是为了解决：热库删除失败，但是冷库插入成功导致的数据不一致的情况
  4. 从热数据库中删除对应的数据

还要考虑定时任务批量操作数据库时的数据量大问题（可以在业务低峰时开始定时任务），假设每次迁移1000条数据

  1. 定时任务开始时使用 update 语句给符合条件的数据添加WaittingForMove标识
  2. 找出前1000条待迁移的数据（ColdFlag=WaittingForMove）
  3. 在冷数据中保存一份数据
  4. 从热数据库中删除对应的数据
  5. 循环执行2～4

还要考虑并发性的问题：假如每次定时任务执行时，单线程无法应对，需要使用多线程来应用。

  1. 启动多线程：可以使用一个线程池来进行多线程的操作
  2. 锁争抢，某线程宣布正在操作某个数据，其他线程不能操作它

2. 查询分离​

2.1 案例背景​

itemdesc
工单表数据 10_000_000关联13个表，查询花费13s
工单 详情页关联数据表多，5s

2.2 查询分离概念​

查询分离即每次写数据时保存一份数据到其他的存储系统中，用户查询数据时直接从中获取数据，适用场景：

  • 数据量大
  • 查询数据的响应效率很低
  • 所有写数据请求的响应效率尚可
  • 所有数据任何时候都可能被修改和查询，针对冷热分离的缺点来说的

2.3 具体实现​

（1）如何触发查询分离

  • 修改业务代码，在写入常规数据后同步维护查询数据； 写操作变慢
  • 修改业务代码，在写入常规数据后，异步更新查询数据； 更新有延迟，可能查询到过时数据
  • 监控数据库日志binlog，有数据变更，触发数据的更新操作； 引用中间件复杂，更新有延迟

这里业务上综合了客户：1.希望写操作响应及时 2.业务代码并不复杂 两个方面考虑使用方案二，业务代码实现异步更新读库

（2）如何实现查询分离

  • 启动线程来维护查询数据；问题：1.线程不能无限创建，可以使用线程池 2.创建失败，如何重试？ 3.并发问题
  • 引入消息中间件；问题： 1.MQ 选型 2.MQ 问题：消息丢失、重复消费

消息中间件的作用：1.解耦两个服务 2.提高主服务写操作的吞吐量

（3）查询数据如何存储？ES 或者 MongoDB，可以从团队对技术的熟悉程度来考虑。

（4）查询数据如何使用？可用调用ES提供的 API，注意维护写库过程中进行查询，可以导致数据不一致的情况。

  • 不允许用户查询
  • 给用户提示，目前查询到的数据可能是2秒前的数据，可以尝试刷新一下。

（5）历史数据迁移

可以全部加上标识

2.4 Elasticsearch 注意事项​

（1）如何使用 ES 设计表结构？
ES 是基于索引的设计，无法进行关联查询，所以维护写库时需要把每条主数据及关联子表的数据全部整合在一条记录中

（2）ES 的存储结构 ES 是一个分布式的查询系统，它的每一个节点都是一个基于 Lucene 的查询引擎。

ES是基于倒排索引的，对于有结构文档（即包含多个字段）

3.分库分表​

Client 模式与 Proxy 模式的优缺点

模式优点缺点
Proxy多语言、资源消耗解耦、不需要消耗客户端资源、升级方便多一层服务调用、线上调试困难、增加了运维成本
Client少一层调用、运维成本少单语言、升级不方便

3.3 分表分库实现思路​

  1. 使用什么字段作为分片主键？

原则：

  • 数据尽量均匀分布
  • 跨库操作尽可能少
  • 所选字段的值不会变

以订单为例，一般有 user_id, order_id 等字段，这里选取 user_id 作为分片主键，主要考虑是
用户查询订单操作较多，使用 user_id 分片，就可以保证这个操作在一个库的一个表中查询。

  1. 分片策略是什么？
  • 根据范围分片
  • 根据 Hash 值分片
  • 根据 Hash 值及范围混合分片

一般建议拆分为2n 个表，这样数据迁移时把原来的每一张表拆一半出来组成新表，这样数据迁移量就小了。
方案根据 user_id hash 值按32取模，把数据分到32个库中，每个数据库再分成16个表。
假如每天订单量为 10_000_000，则每个库是增 10_000_000 / 32 = 31.25万，每个表日增 31.25/16 = 2w 数据，3年后每个表的数据量就是
2 365 3 = 2000万左右，仍在可控范围内。

  1. 业务代码如何修改？
  • 单体服务分表分库很麻烦，因为涉及到的关联查询太多
  • 互联网架构中，基本不使用外键
  • 尽量避免跨库、跨表查询
  1. 历史数据如何迁移？

基本思路：旧架构继续运行，存量数据直接迁移，增量数据监听binlog，然后通过 canal 通知迁移程序迁移数据，等到新的数据库拥有全量数据且校验通过后再逐步切换流量到新架构。

1）上线 canal，通过 canal 触发增量数据的迁移 2）迁移数据脚本测试通过后，将老数据迁移到新的分表分库中 3）注意迁移增量数据与迁移老数据的时间差，无遗漏 4）此时新的分表分库已经拥有全量数据了，可以运行数据验证程序验证

  1. 未来的扩容方案是什么？
  • 使用2的 n 次幂的方案，每次可以扩容2倍
  • 数据迁移，同上历史数据迁移

总结​

1亿订单量，32个库，16个表的压力下，数据库可以做到20ms 响应时间内

不足：

1）复杂查询慢：查询需要跨订单数据库进行组合结果集；解决办法是使用查询分离，这里采用的 ES+HBase，ES 返回查询结构到页面，用户点击详情后，再到 HBase 中去查询。 2）增量数据迁移的高可用性和一致性： 3）短时订单量大爆发：数据量大可以使用数据库来解决，瞬时流量大如秒杀等场景，就要依靠缓存来解决。

缓存层场景实战​

读缓存​

业务场景：将十几秒的查询请求优化成毫秒级​

商城场景下的商品详情页的数据：商品的图片、介绍、规格、评价、商品推荐。最近成交情况、优惠活动。
查询请求执行多个 SQL 语句，导致需要在5w 的数据量的情况下还需要十几秒才能打开详情页
使用的方案：1.将大部分的详情数据缓存起来 2.将商品成交数据异步加载（即用户打开商品详情页后再加载最近的成交数据）

缓存选型​

何时存储数据​

旁路缓存： 缓存击穿：热点 key 过期或者不存在，解决：加锁 缓存穿透：数据库也不存在的数据，每次查询都会走数据库，解决：缓存空值或者布隆过滤器 缓存雪崩：Redis 宕机或者数据大面积过期，解决：高可用+TTL 随机值 缓存预热：业务访问量低时，将预热的数据保存到缓存中，这样流量大的时候，就可以直接从缓存中读取数据。

如何更新缓存​

1.先更新缓存，再更新数据库：直接排除， 数据库回滚，缓存无法撤销 2.先删除缓存，再更新数据库：有问题，数据库更新耗时多，在此期间请求访问，会重置缓存，造成数据不一致的情况，会持续到缓存 TTL 3.先更新数据库，再更新缓存： 双写并发问题 4.先更新数据库，再删除缓存：也有读写并发问题，数据库更新后删除缓存成功前，读访问就会读取到脏数据，但是因为删缓存操作耗时比较短，综合来看，可以忽略不计。 5.双删：先删除缓存，再更新数据库，再删除缓存：还是会出现4中的问题 6.延迟双删：
