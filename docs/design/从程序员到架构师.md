#### 1.3 冷热分离

冷热分离就是在处理数据时将数据库分为冷库和热库，冷库存放那些走到终态、不常使用的数据，热库存放还需要修改，经常使用的数据。\
使用冷热分离的情形：

1. 数据走到终态后只有读没有写的需求，如订单完结状态。
2. 用户能接受数据分开查询，比如有些电商默认只允许查询3个月内的订单数据。如果要查询3个月前的订单，还需要访问其他的页面。

#### 1.4 冷热分离实现思路：冷热分离都使用 mysql

遇到的问题：

1. 如何判断一个数据是冷数据还是热数据？
2. 如何触发冷热数据分离？
3. 如何实现冷热数据分离？
4. 如何使用冷热数据？
5. 历史数据如何迁移？

判断一个数据冷热数据：可以使用表中某个字段，可以是时间、状态，也可以结合时间和状态 \
如何触发冷热数据分离：\

- 每次涉及到订单修改时，判断是否进行冷热分离数据处理；根据状态区分冷热数据时可以使用。
- 可以通过监听数据库变更日志 binlog + 另起服务监控 + 消息队列；业务代码复杂不适合修改，并且不根据时间区分冷热数据
- 定时扫描数据库；按照时间区分冷热数据时使用。

如何分离冷热数据？

其实质是删除热数据，并插入对应的冷数据。这是一个事务特性的操作，但是因为冷热库分离，所以无法依赖数据库的事务特性。解决办法是最终一致性的：

1. 将热库待删除的数据添加一个标识`WaitForMove`，然后将其删除。
2. 找出所有待迁移的数据（通过 WaitForMove）。这一步保证之前的数据迁移失败，可以进行补偿操作
3. 在冷库中插入一份数据，这个数据要有幂等性判断。这一步是为了解决：热库删除失败，但是冷库插入成功导致的数据不一致的情况
4. 从热数据库中删除对应的数据

还要考虑定时任务批量操作数据库时的数据量大问题（可以在业务低峰时开始定时任务），假设每次迁移1000条数据

1. 定时任务开始时使用 update 语句给符合条件的数据添加`WaittingForMove`标识
2. 找出前1000条待迁移的数据（ColdFlag=WaittingForMove）
3. 在冷数据中保存一份数据
4. 从热数据库中删除对应的数据
5. 循环执行2～4

还要考虑并发性的问题：假如每次定时任务执行时，单线程无法应对，需要使用多线程来应用。

1. 启动多线程：可以使用一个线程池来进行多线程的操作
2. 锁争抢，某线程宣布正在操作某个数据，其他线程不能操作它

### 2. 查询分离

#### 2.1 案例背景

| item | desc |
| --- | --- |
| 工单表数据 10_000_000 | 关联13个表，查询花费13s |
| 工单 详情页|  关联数据表多，5s |

#### 2.2 查询分离概念

查询分离即每次写数据时保存一份数据到其他的存储系统中，用户查询数据时直接从中获取数据，适用场景：

- 数据量大
- 查询数据的响应效率很低
- 所有写数据请求的响应效率尚可
- 所有数据任何时候都可能被修改和查询，针对冷热分离的缺点来说的

#### 2.3 具体实现

（1）如何触发查询分离

- 修改业务代码，在写入常规数据后同步维护查询数据；          写操作变慢
- 修改业务代码，在写入常规数据后，异步更新查询数据；        更新有延迟，可能查询到过时数据
- 监控数据库日志binlog，有数据变更，触发数据的更新操作；    引用中间件复杂，更新有延迟

这里业务上综合了客户：1.希望写操作响应及时 2.业务代码并不复杂 两个方面考虑使用方案二，业务代码实现异步更新读库

（2）如何实现查询分离

- 启动线程来维护查询数据；问题：1.线程不能无限创建，可以使用线程池 2.创建失败，如何重试？ 3.并发问题
- 引入消息中间件；问题： 1.MQ 选型 2.MQ 问题：消息丢失、重复消费

消息中间件的作用：1.解耦两个服务 2.提高主服务写操作的吞吐量

（3）查询数据如何存储？ES 或者 MongoDB，可以从团队对技术的熟悉程度来考虑。

（4）查询数据如何使用？可用调用ES提供的 API，注意维护写库过程中进行查询，可以导致数据不一致的情况。

- 不允许用户查询
- 给用户提示，目前查询到的数据可能是2秒前的数据，可以尝试刷新一下。

#### 2.4 Elasticsearch 注意事项

（1）如何使用 ES 设计表结构？\
ES 是基于索引的设计，无法进行关联查询，所以维护写库时需要把每条主数据及关联子表的数据全部整合在一条记录中

（2）ES 的存储结构
ES 是一个分布式的查询系统，它的每一个节点都是一个基于 Lucene 的查询引擎。

| Lucene | mysql |
| --- | --- |
| Index | database |
| Type | table |
| Document | row |
| Field | column |
| Hit | result |

ES是基于倒排索引的，对于有结构文档（即包含多个字段）
