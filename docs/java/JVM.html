JVM

虚拟机种类​

HotSpot 虚拟机：OracleJDK 和 OpenJDK 使用的虚拟机；它的主要优势是：

  • 准备式内存管理：虚拟机知道内存中某个位置的数据具体是什么类型，是基本数据类型还是引用地址
  • 热点代码探测技术：一个方法频繁调用或者方法中有效循环次数多，将会分别触发标准即时编译和栈上替换编译(On-Stack Replacement，OSR)行为

内存管理​

运行时数据分区​

程序计数性 PC​

线程私有，唯一不会OOM 的区域。指向下一个执行的字节码指令，可用于分支、跳转等流程中

虚拟机栈与本地方法栈​

虚拟机栈线程私有，每个方法执行会在其中添加一个栈帧，其中存储的是局部变量表(方法参数和局部变量)、操作数栈(方法执行过程中的临时操作数，实现算术运算等)、动态连接（指向运行时常量池中该方法的引用）、方法出口等。 虚拟机栈定义了两个异常：

  • 请求的栈深度大于了虚拟机允许的深度，StackOverFlowError
  • 如果栈动态扩展导致内存申请失败，会 OutOfMemoryError

本地方法栈与虚拟栈一致，不同就是它存储的是 native 方法的栈帧

堆​

堆线程共享，JVM启动时创建；几乎所有的对象实例和数组都在堆上分配（例外如逃逸分析，栈上分配） 当堆上没有内存足够分配，并且堆不不能再扩展时，报 OutOfMemoryError

方法区​

线程共享区，存储已被加载的类信息，常量，静态变量，JIT 编译后的代码缓存等。方法区也有回收，主要是针对常量池的回收和类型的卸载，如果方法区无法满足内存分配，报 OutOfMemoryError

运行时常量池​

常量池是方法区的一部分，Class 文件中有常量池表，用于存放编译期生成的各种字面量与符号引用，这部分内容将在类加载后存放到方法区的运行时常量池。并非 Class 文件预置的数据才能进入常量池，例如调用 String.intern()方法就可以使字符串进入方法区，所以它在内存不足分配时，也会报 OutOfMemoryError

直接内存​

特指 NIO 中的 ByteBuffer，它是由 OS 分配的内存，然后通过java 堆中的对象作为这块内存的引用进行操作。它不受堆内存大小的限制，但要受宿主机主内存的限制。也会报 OutOfMemoryError

对象结构​

对象的创建方式​

创建一个对象的方式：

  • new
  • 反射
  • clone
  • 反序列化

java 中对象的内存的大小是确定，分配方法则有两种

  • 指针碰撞，适用堆内存规整->垃圾回收器整理功能->Serial、ParNew
  • 空闲列表，适用堆内存零散->CMS

分配内存时的并发问题：

  • CAS+失败重试
  • 本地线程分配缓冲（TLAB），本地缓冲用完了，才会同步锁定分配公共内存

内存分配完成后，将内存区域除对象头外赋默认值，即类型零值

对象的内存结构​

  • 对象头
    • Markword，即对象运行状态信息
    • 类型指针，指向无数据（方法区的 Class 对象）、数组的长度
  • 实例数据部分，包括父类的变量和子类的变量
  • 数据填充部分，内存起始地址需要是8的整数倍，即每个对象的大小必须是8的整数倍

对象的访问定位​

有两种主要方式，HotSpot 使用的是直接指针访问

  • 句柄访问：堆中有单独的句柄池，存储对象的实例数据地址（堆）和类型地址（方法区），这种方式下访问对象：栈中的引用->句柄池中的句柄->实例数据内存和类型地址，优点是当垃圾回收进行对象移动时，不需要修改栈上的引用
  • 直接指针访问：栈引用直接指向堆中的对象实例数据，这样的好处是少一次指针访问。但是对象中需要存储类型信息地址

OOM 分析​

掌握：

  1. 从异常信息分析出是哪个区域出现 OOM
  2. 分析什么样的代码导致该区出现 OOM

堆溢出：java.lang.OutOfMemoryError: Java heap space；注意线上环境使用-XX:+HeapDumpOnOutOfMemoryError在堆内存溢出时自动保存内存映像。然后使用内存映像分析工具（如 Eclipse Memory Analyzer）对堆转储快照分析。

  1. 确认是内存泄漏还是内存溢出，区别在于申请的对象是否是必要的。
  2. 如果是内在泄漏，进一步通过工具查看泄漏对象到 GC Roots 的引用链。从而定位到对象创建的位置，泄漏的具体代码。
  3. 如果是内存溢出，两个思路：（1）查看机器物理内存还有无向上调整的空间 （2）查看对象的持有状态时间，生命周期时间，存储结构等来减少对象的内存消耗

经典垃圾回收器​

Serial 收集器：进行垃圾收集时，必须暂停其他所有工作线程，Stop The World。

ParNew 收集器：多线程版本的 Serial 收集器，与 CMS 搭配使用，它应用于年轻代，而 CMS 应用于老年代

Parallel Scavenge收集器：它更关注达到可控的控制量，即用户线程执行时间/用户线程执行时间+垃圾回收线程时间

Serial Old收集器：它是 Serial 垃圾收集器的老年代版本，特性一致

Parallel Old收集器：Parallel Scavenge 收集器的老年代版本

CMS 收集器：Concurrent Mark Swap，最短回收停顿时间，用于老年代，采用的是标记-清除算法。其作用过程分四个部分：

  • 初始标记：只标记 GC Roots 直接访问到的对象，耗时较短，要暂停用户线程。STW
  • 并发标记：从初始标记到的对象遍历到整个对象图，耗时长，但是可以与用户线程并发进行。
  • 重新标记：修正并发标记期间，因用户程序运行行为导致标记产生变动的一部分对象，需要重新标记，不能与用户线程并发进行。STW
  • 并发清除：根据标记清除死亡对象，不需要移动存活对象，所以可以和用户线程并发进行。

G1收集器：不再回收分代，面向区域 Region 的回收。G1收集器将堆空间划分为多个 Region，由 Region 来组成年轻代或是老年代，收集垃圾时可以优先回收回收效率更高的 Region。
